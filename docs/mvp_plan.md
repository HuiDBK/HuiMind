# HuiMind MVP 计划（通用学习平台 + AI 学习搭子 + career 首发 / MySQL）

## 1. 项目目标

### 1.1 首版目标

HuiMind 首版 MVP 的目标不再只是验证单一求职场景，而是先做出一个可复用的 `通用学习平台底座`，并在这个底座上交付首个高价值官方场景 `career`。首版需要验证两层价值：

- 平台层价值：用户上传资料后，是否愿意持续使用知识问答、薄弱点追踪、复习任务和 AI 学习搭子。
- 场景层价值：在 `career` 场景下，用户是否愿意为简历诊断和模拟面试持续使用产品。

### 1.2 本次验证的问题

- 用户是否会把 HuiMind 当作通用学习空间，而不是一次性工具。
- AI 学习搭子是否能提升连续使用率和复习完成率。
- 通用学习底座是否足够稳定，能支持 `general` 与 `career` 两类场景共享能力。
- `career` 是否适合作为首个官方高付费意愿场景。

### 1.3 首版范围修正

- 关系数据库首版统一采用 `MySQL`。
- 首版仍以 `Web-only` 交付，不引入飞书、QQ、Webhook 或 Bot。
- 首版官方场景包含：
  - `general`：通用学习平台默认场景
  - `career`：首个垂直官方场景
- `Redis` 与 `Celery` 纳入首版必备基础设施，不再作为后续预留。

## 2. MVP 范围

### 2.1 必做能力

- 登录与个人学习空间初始化
- 场景入口与切换
  - 默认通用场景 `general`
  - 官方场景 `career`
- 知识库管理
  - 支持 PDF、TXT、Markdown 上传
  - 支持 JD 文本录入或 URL 导入
  - 支持按场景隔离文档
- 文档解析、切块、向量化、入库
- 通用 RAG 问答
  - 基于当前场景资料回答
  - 返回引用来源
  - 资料不足时拒答
- 薄弱点提取与记录
- 复习任务生成与完成状态更新
- AI 学习搭子
  - 自定义名字
  - 选择人设风格
  - 具备跨会话连续感
  - 基于学习记录生成主动提醒文案
- Dashboard
  - 最近资料
  - 薄弱点
  - 待复习任务
  - AI 搭子状态
- `career` 场景能力
  - 简历诊断
  - 模拟面试

### 2.2 明确不做

- 飞书集成
- QQ 集成
- 支付、订阅、团队版
- 战队、排行榜、知识对战
- 多用户共享知识库
- 场景广场
- 用户自建场景发布与 Fork
- 原生 App 客户端
- 复杂游戏化系统
  - 成就体系
  - 技能树
  - 闯关模式

### 2.3 MVP 边界原则

- 先把 `通用学习底座 + AI 学习搭子 + career 首发场景` 做通。
- 首版要能支持后续扩第二、第三官方场景，但不做完整平台生态。
- Redis 与 Celery 首版必须投入使用，用于会话、缓存、异步解析和复习调度。

## 3. 目标用户与核心路径

### 3.1 目标用户

- 有明确学习资料、希望建立持续学习闭环的用户
- 应届生与求职者
- 有长期自学需求、希望获得记忆感和督促感的用户

### 3.2 用户核心痛点

- 资料很多，但缺乏统一沉淀和可追踪使用方式
- 学过的内容不会沉淀，容易反复遗忘
- 与 AI 的对话没有连续感，每次都像重新开始
- 求职准备缺少针对性反馈

### 3.3 核心用户路径

通用学习路径：

`登录 -> 进入 general -> 上传资料 -> 提问 -> 识别薄弱点 -> 生成复习任务 -> 与 AI 学习搭子互动`

求职场景路径：

`登录 -> 进入 career -> 上传简历/JD -> 生成简历诊断 -> 发起模拟面试 -> 查看薄弱点 -> 完成复习任务`

### 3.4 首版用户价值

- 用一个平台统一承载学习资料、问答、记忆、复习和陪伴感
- AI 学习搭子让产品具备连续感和人格感，而不是纯工具
- 在 `career` 场景下，把资料、诊断、面试和复盘串成闭环

## 4. 功能拆分

### 4.1 账户与场景子系统

- 登录
- 获取当前用户信息
- 初始化默认场景
- 获取可用官方场景列表

### 4.2 资料子系统

- 上传学习资料
- 录入 JD 文本或 URL
- 查看文档列表与处理状态
- 按场景过滤资料

### 4.3 问答子系统

- 基于当前场景资料进行问答
- 回答附带引用来源
- 记录问答上下文和问答历史

### 4.4 记忆与复习子系统

- 薄弱点抽取
- 复习任务生成
- 复习完成与再次安排
- 复习到期调度

### 4.5 AI 学习搭子子系统

- 人设配置
- 名字配置
- 搭子对话
- 结合上下文和学习记录生成个性化反馈
- 生成提醒文案和复习鼓励文案

### 4.6 `career` 场景子系统

- 简历诊断
- 模拟面试 Session
- 面试逐题评分与总结

### 5.1 登录页

## 6. 里程碑

### Phase 0：工程骨架与基础设施

- 搭建 FastAPI 与 Next.js 基础工程
- 打通 MySQL、Redis、Chroma、Celery、Celery Beat
- 明确统一响应结构、配置管理、日志链路

### Phase 1：通用学习平台底座

- 完成资料上传、解析、切块、向量入库
- 完成场景隔离和通用知识问答
- 完成引用展示和拒答策略

### Phase 2：记忆与复习

- 完成薄弱点抽取
- 完成复习任务生成
- 使用 Celery Beat 扫描到期复习任务

### Phase 3：AI 学习搭子

- 完成搭子配置
- 完成搭子对话
- 完成学习记录驱动的个性化反馈与提醒文案

### Phase 4：`career` 场景

- 完成简历诊断
- 完成模拟面试
- 将场景结果回流到薄弱点与复习链路

### Phase 5：前端串联与内测

- 串通首页、场景页、文档页、问答页、搭子页、复习页、career 页面
- 增加空状态、错误态、样例数据
- 完成首轮内测与反馈收集

## 7. 交付标准

### 7.1 产品标准

- 用户可以在 10 分钟内走通至少一条完整学习路径
- 用户能清楚理解资料、问答、薄弱点、复习任务之间的关系
- 用户能感知 AI 学习搭子的连续感和个性化

### 7.2 研发标准

- MySQL、Redis、Celery、Chroma 全部落地可运行
- 通用学习路径与 `career` 路径均可跑通
- 关键数据具备结构化存储和场景隔离能力

### 7.3 成功标准

- 用户愿意在 `general` 场景持续上传资料和提问
- 用户认为 AI 学习搭子不是装饰，而是能提升陪伴感和复习动力
- `career` 场景的简历诊断和模拟面试被认为有明确价值
- 代码结构可继续扩展更多官方场景

## 8. 技术与产品取舍

### 8.1 采用 MySQL 的原因

- 与当前仓库配置和开发习惯更一致
- 足以支撑首版结构化数据管理
- 便于在 MVP 阶段快速落地和维护

### 8.2 先做通用学习平台的原因

- 通用能力是所有官方场景的底座
- 先沉淀资料、问答、记忆、复习、搭子，后续扩场景成本更低
- 能避免首版做成只服务求职的一次性工具

### 8.3 将 AI 学习搭子纳入首版的原因

- 搭子是产品差异化的重要来源
- 搭子能承接长期记忆和学习提醒，直接影响留存
- 先做最小人格化和连续感，比后期硬插入更合理

### 8.4 Redis 与 Celery 首版落地的原因

- Redis 用于会话上下文、热点缓存和任务状态协同
- Celery 用于文档解析、向量化、诊断等异步任务
- Celery Beat 用于复习任务调度和周期扫描

## 9. 后续扩展预留

以下能力仍作为后续扩展，不纳入首版承诺：

- 飞书和 QQ Bot
- 支付和团队版
- 场景广场
- 用户自建场景发布
- 社交化和重游戏化玩法
