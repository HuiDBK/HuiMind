## AI 伴学平台 Agent 架构

<br />

***

### 定性

单 Agent + 动态 Scene Config。三个官方场景执行流程相同，差异在配置不在逻辑，用多 Agent 是过度设计。

***

### 核心概念

<br />

| 概念       | 是什么                                               | 存在哪                     |
|------------|------------------------------------------------------|----------------------------|
| Tool       | Agent 的执行能力，LLM 决定调不调                     | 代码                       |
| Skill      | 场景领域知识 + 工具调用策略                          | 提示词，数据库读取         |
| Scene      | Tool + Skill + 知识库 + 评分规则的配置容器           | 数据库                     |

<br />

***

### 六个 Tool

```
search_knowledge      混合检索知识库（BM25 + 向量）
generate_quiz         出题巩固
update_weakness       更新薄弱点画像
rubric_evaluate       长文本结构化评分（维度由 eval_rubric 决定）
schedule_review       安排艾宾浩斯复习
web_search            联网补充时政热点

```

同一个 Tool 在不同场景行为不同。`rubric_evaluate` 在求职场景是 STAR 四维，在考公场景是申论五维，区别来自 `eval_rubric` 配置，不是两个 Tool。

***

### 四个场景

**求职助手** 工具集：检索 + 评分(STAR) + 出题 + 画像 Skill 重点：简历逐行诊断、面试难度循序渐进、STAR 法则强化

**考研备考** 工具集：检索 + 出题 + 画像 + 番茄钟 Skill 重点：错题 DNA 追踪、真题解析归纳规律、战队互动

**考公备考** 工具集：检索 + 评分(申论) + 爬虫 + 画像 Skill 重点：申论五维批改、时政热点注入、行测解题路径

**通用场景** 工具集：检索 + 出题 + 画像（最小集） 三个职责：

- 新用户冷启动，引导上传资料并推荐官方场景
- 用户自建场景的底座，fork 后加配置变新场景
- 公共知识库，跨场景知识迁移的中转站

***

### Sub-Agent 引入条件

现在不需要。以下情况出现时再引入：

```
一个请求需要多专家并行
  例：模拟考 = 出题专家 + 评分专家同时工作

某场景内部逻辑需要自己的 ReAct 循环
  例：申论批改内部 → 检索时政 → 分析结构 → 评分 → 生成示例
  复杂到无法压缩成单个 Tool，包装成 Sub-Agent

跨场景协作任务出现
  例："结合我的考研薄弱点和实习准备，安排这个月计划"
  这时才需要 Orchestrator

```

***

### 演进路径

```
MVP
  单 Agent + 动态 SceneConfig
  场景配置真正驱动 Agent 行为

V2（场景广场）
  Skill Registry 独立化
  用户自建场景 = 从 Registry 选 Skill 组合
  不用手写 prompt，降低门槛

V3（模拟考 / 复杂任务）
  Sub-Agent as Tool
  Orchestrator 处理跨场景并行
  现有 Skill 直接复用为子节点

```
